Fable usage bar — 4 concepts

Rendered at true HUD pixel size, on a checkerboard. Left = regular, middle = compact, right = 2× zoom of the regular card.
A — Stacked third row · 200×124 regular, 120×104 compact (+12px height)
Adds a fourth grid row that mirrors the Weekly row exactly. Recommended. Nothing gets narrower, every bar keeps its full width, and the label/bar/pct rhythm is unchanged — a reader who already knows the card learns nothing new. The cost is 12px of window height, which for a HUD is nothing. Row-gap drops 6px → 5px so the growth stays under the shadow of the existing card.

B — Split weekly row · 200×112, 120×92 (no height growth)
Both weekly limits share the existing row, side by side. Free in height, but each bar drops from ~120px to ~46px of track — at that width a 1% change moves the fill by half a pixel, so the bar stops being readable as a gauge and becomes decoration next to the number. In compact it's ~18px of track, which is unusable. Only worth it if window height is a hard constraint.

C — Dual rail, one "Weekly" label · 200×118, 120×98 (+6px height)
Two 3px rails stacked inside the weekly slot, sharing one "Weekly" label — which is honest, since both windows reset at the same instant. Half the height cost of A and the bars keep near-full width. The tradeoff is the 3px rails and the 9px ALL/FBL keys: it reads as a denser instrument panel, less calm than the current card. Good if you like the tighter look.

D — Fable as a chip, no bar · 200×112, 120×92 (no height growth)
Weekly keeps its full-width bar; Fable rides beside it as a violet chip showing only the number. Zero layout change, zero height cost, and the weekly gauge stays readable. The catch: Fable has no bar, so you read it instead of glancing at it — fine if Fable is a secondary number you check occasionally, wrong if it's the limit you actually run into first.
